## Key things verified (not taken on faith)

**Verification gate still prevents coincidental/broken auto-accept (ac-03/ac-04).**
The Accepted branch requires `integrity_holds() && parent_unique && confidence ≥ 0.6` —
three conjuncts. The boolean trap (case c) is killed two ways independently: the parent
`gadget_flags.active` is non-unique (2 distinct / 4 rows → `parent_unique = false`), and
its `parent_key_likeness` drags confidence to ~0.27, below the 0.6 floor. The orphan
near-miss (case b) is rejected on orphan count. Surrogate-id overlap is defused by
`name_similarity` returning 0.0 for a child column literally named `id` (its own PK, not
a reference). The `only_the_holding_fk_auto_accepts` test asserts the accepted set is
exactly `{orders.customer_id → customers.id}`. Sound.

**Read-only against DuckDB (choice 0001/0014).** Confirmed empirically: SHA-256 of the
DuckDB file is identical before and after a relate run, no WAL left behind. Source-level:
`relate.rs` issues only `query_row`/`prepare` SELECTs; the only DDL strings are inside
`constraint_ddl` (an emitted artifact String), never executed against the live
connection. Querying-for-discovery, not pipeline execution — choice 0014's read/emit
split holds.

**Descriptor is Frictionless-conformant with FK custom props (ac-05).** Validated via the
in-suite jsonschema validator against the vendored profile, with the FK entries and
custom `x-dovetail*` properties present. The profile permits custom keys; validation is
clean.

**Bin took no direct duckdb dependency.** `crates/dovetail/Cargo.toml` depends only on
`dovetail-core` (default-features off) and `anyhow`. duckdb correctly moved from
`[dev-dependencies]` to `[dependencies]` in `dovetail-core` (with a choice-0014 comment),
so it's a transitive-via-core runtime dep — the bin stays thin, and `run_path` keeps the
duckdb-touching code in core.

## Non-blocking observations

**[LOW] No explicit parent minimum-cardinality floor.** The auto-accept gate trusts
`parent_unique` (distinct == total) plus naming-driven confidence. A parent that is
*accidentally* unique but low-cardinality (e.g. a 2-row, 2-distinct table) clears
`parent_unique`; only `parent_key_likeness` (name-driven) then keeps confidence under
0.6. The fixture's boolean trap happens to also be non-unique, so this exact corner
isn't exercised. It is arguably a *correct* accept (the edge genuinely verifies), and
naming gates it in practice — but the safety story leans on the confidence threshold
rather than an explicit cardinality floor for this case. Worth a line in the
status-assignment logic or a fixture case in a follow-up spec; not a defect in this PR.

**[INFO] Connection opens read-write.** `Connection::open(path)` is read-write mode; the
read-only guarantee is enforced by discipline (only SELECTs issued), confirmed
empirically. A `Connection::open_with_flags(..., READ_ONLY)` would make the boundary
structural rather than behavioural and would also let relate run against engine-held DBs.
Optional hardening, not required by any AC.
