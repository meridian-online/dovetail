## Findings

All findings are LOW. None blocks merge.

### [LOW] CLI uses the shape detector, not the canonical finetype-guided one
**Where:** `crates/dovetail/src/main.rs:41`, `crates/dovetail/Cargo.toml`
**What:** ac-04 names `FinetypeGuidedDetector` as survey's canonical detector, but
the CLI binary constructs `ShapeHeuristicDetector`, and the `dovetail` crate pins
`dovetail-core` with `default-features = false` so the finetype arm never compiles
into the binary. This is defensible and the code comment says so: with no model dir
the finetype-guided detector degrades to *exactly* the shape detector's structural
output, so the CLI's structure results are identical today, and avoiding the candle
stack keeps the CLI build light. But it means the canonical detector chosen in ac-04
is not the one the shipped binary runs — the semantic typing that justified the pick
(for ac-06 schemas and later relate) is absent from the CLI path. Acceptable for the
MVP; worth a follow-up to wire `FinetypeGuidedDetector::from_env()` behind a feature
once model artifacts exist. The decision note's own "strictly dominates once artifacts
are present" framing already anticipates this.

### [LOW] ac-11's "duplicate-key" half (JSON) is unexercised
**Where:** `detect/shape.rs` (json/ndjson paths), `tests/fixtures/`
**What:** Duplicate detection covers delimited headers (CSV/TSV) only. JSON/NDJSON
go through `serde_json`, which silently collapses repeated keys within an object
before detection sees them, and there is no JSON-duplicate-key fixture. ac-11's text
says "duplicate-key / duplicate-column"; ac-01 only mandated the CSV dup-columns
fixture, so this is within the specified corpus — but the duplicate-*key* surface of
ac-11 is not actually tested. Low risk for the SQL-native MVP (the load readers
resolve their own dup behaviour); flag a JSON-dup-key fixture for the hardening
follow-up so the "never silently drop" guarantee holds for JSON too.

### [LOW] All fixtures share the table name `data`
**Where:** `survey.rs::resource_name`, every fixture file named `data.<ext>`
**What:** `resource_name` derives the table/resource name from the file stem, which
is `data` for every fixture, so the CLI emits `CREATE OR REPLACE TABLE "data"` for
all inputs. Harmless in the corpus (each emission is standalone) and the sanitiser
is correct, but surveying two real files with the same stem would collide on table
name. Not in scope for this spec's fixtures; note for when survey writes multiple
descriptors to one output tree.

## Honest assessment

This is a clean, well-tested implementation that does what the spec asked and is
honest about what it doesn't yet prove. The two gate ACs that matter most — the
round-trip (ac-07) and the detection bar (ac-10) — have loud, runnable evidence,
and the emit-don't-execute boundary is genuinely held rather than asserted. The
degraded-mode limitation (the finetype arm matched rather than beat the structural
baseline because the corpus is flat and no model artifacts are wired) is the one
real soft spot, and it is disclosed exactly where a reader would look: the ac-04
decision note and a distill memo. That is the right way to carry a known gap, not
a reason to withhold approval.

The three findings are all MVP-appropriate deferrals already implied by the spec's
own scoping (SQL-native only, finetype as future enrichment). The CLI/canonical-detector
divergence (finding 1) is the one I'd most want tracked as a follow-up, because it is
the gap between "the detector we picked" and "the detector that ships" — but it is
behaviourally a no-op today and the code says so. APPROVE.
