## The safety gate is sound — I tried to break it

The verification gate (ac-03) is the load-bearing claim: an edge auto-accepts only if referential integrity holds (zero orphans) AND the parent is unique/PK-like. I ran four adversarial probes against `discover()` beyond the fixture:

- **Surrogate-id coincidence** (the brief's main worry). A decoy unique parent (`tickets.id` 1..6) that fully covers a child's `user_id` range 1..5. It never becomes a candidate: `name_similarity` returns 0.0 unless the child is FK-named for that parent or column-name-equal, so the decoy is pruned *before* any overlap query. Only the truthful `orders.user_id -> users.id` accepted. **Holds.**
- **Child column named bare `id`.** `name_similarity` hard-returns 0.0 for `cc == "id"` (relate.rs:326) — a table's own PK can't masquerade as an FK, killing the "two tables both keyed 1,2,3" trap at the source. **Holds.**
- **One orphan against a strong, unique, well-named parent** (orphan_rate exactly at the 5% `ORPHAN_TOLERANCE` boundary). Routes to Suggested, never Accepted. Accept requires `integrity_holds()` which is strict `orphan_count == 0` (relate.rs:62, 275); the tolerance band only softens Suggested-vs-Rejected, it is never a path to Accepted. **A broken reference cannot slip through to accepted.**
- **Non-unique parent with zero orphans + high confidence** (the boolean trap generalised to a duplicated id). Never accepts — `parent_unique` is a hard conjunct of the accept condition. **Holds.**

The confidence score multiplies name/overlap by `parent_key_likeness`, so a perfect name+overlap match against a non-key parent stays low-confidence — a second, independent brake on the boolean trap on top of the hard `parent_unique` gate. Good defence in depth.

**Edge cases the fixture misses (limitations, not defects):**
- **Self-referential FKs are undiscoverable.** `discover()` skips all same-table pairs (relate.rs:111), so `employees.manager_id -> employees.id` is never a candidate (confirmed by probe). Out of the spec's stated scope, but worth a one-line known-limitation note on card 0002.
- **Composite / multi-column keys** are not generated — candidates are single-column only. The `ForeignKey.fields` type is a `Vec`, so the descriptor shape is composite-ready, but discovery never populates more than one. Also out of scope; flag as known.
- **NULL handling is correct.** Orphan and overlap queries filter `IS NOT NULL` on both sides, so the `NOT IN (… NULL …)` three-valued-logic trap is avoided and NULL children are correctly excluded from FK membership.

## Read-only boundary (choice 0001/0014) — holds, with a hardening nit

Every SQL statement relate issues against the connection is a `SELECT`/`count` read (`query_row`/`query_map`/`prepare`, relate.rs:151–239). The only `ALTER` is in `constraint_ddl()`, a string-emitting function — never executed against the connection. Execution stays an emitted artifact. **choice 0001 intact.**

Nit (non-blocking): `Connection::open(path)` (relate.rs:134) opens read-write. The boundary is held by convention (no write statements), not enforced by the engine. Opening with DuckDB's read-only access mode would make the boundary structural rather than a property of the code never drifting. Worth a follow-up; not a blocker.

## ac-06 deviation — sound resolution, no gap

DuckDB rejects `ALTER TABLE ADD FOREIGN KEY` (CREATE-time FK only), so "the ALTER DDL applies cleanly" is literally impossible to demonstrate in DuckDB. The implementer emits portable `ALTER … ADD CONSTRAINT … FOREIGN KEY` (the legible, standard-SQL-accepted migration artifact) and proves the constraint *holds* via a CREATE-time round-trip: rebuild parent-with-PK + child-with-FK, re-insert the verified data, DuckDB enforces it, the insert succeeds because the edge was verified (relate_discovery.rs:91–101). This is a **stronger** proof than re-running relate's own orphan check — it exercises the engine's enforcement, not dovetail's. The emitted artifact and the proof-of-holding are cleanly separated. Accept the deviation.
