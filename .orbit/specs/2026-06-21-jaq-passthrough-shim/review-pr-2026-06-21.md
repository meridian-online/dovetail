## Summary

The shim does what it claims: `dovetail jaq <program> <file>` runs a jq program
through embedded, pinned jaq-core 3.1.0, emits NDJSON, never rewrites the
program, and is byte-stable. Both parity claims hold under test — self-parity
(ac-05, reproducibility) and the scoped reference parity (ac-06, byte-diff vs
system jq 1.8.2 on the curated corpus). All 8 ACs carry real evidence in the
diff; nothing is asserted on faith.

I ran the suite (`cargo test -p dovetail-core --no-default-features --test
jaq_transform`): **6 passed**. I confirmed the load-bearing ac-06 test
*actually executes* rather than skips — jq 1.8.2 is on this machine, and
`reference_parity_vs_system_jq` reports `1 passed; 0 ignored`. I also forced the
absent-jq branch (empty PATH) and confirmed it prints `ac-06: system jq not
found — skipping reference-parity check` and passes rather than fails. The skip
guard is genuine, not a permanently-dead branch.

## Findings

### 1. ac-04 multi-format input is not exercised through jaq (minor — record, don't block)

The AC text says the corpus should include "JSON, plus at least one of
YAML/TOML/XML to exercise **jaq's** multi-format input." The engine path
(`run_jaq` → `jaq_json::read::parse_many`) only parses **JSON**. The
`yaml_input_converts_and_runs` test converts YAML→JSON *in the test harness*
(`serde_yaml::from_str`) and then feeds JSON bytes to `run_jaq`. So the shim
itself does not ingest YAML/TOML/XML — conversion happens outside it, and TOML/XML
are not touched at all.

This is honest about the NDJSON output guarantee, and the spec's own framing
(survey's `.yaml` rung "will later call" the conversions) suggests format
ingestion lands in a later seam. But the as-shipped capability is narrower than
the AC's literal wording. Not a blocker — the JSON-only engine is sufficient for
this spec's stated goal and the test demonstrably exercises a non-JSON *source* —
but the multi-format-input claim should be tightened in a follow-up (either wire
format conversion into the shim, or reword the AC to "non-JSON source converts to
JSON before jaq"). Worth a note on card 0003.

### 2. ac-08 version stamp is hardcoded (known follow-up, not a defect)

`JAQ_CORE_VERSION` is a hand-maintained constant tracking the Cargo.lock pin.
The code comment and spec both flag auto-derivation from Cargo.lock as a
choice-0013 hardening follow-up. The constant currently matches the lock
(3.1.0, verified), so the stamp is correct today; the only risk is silent drift
if the pin bumps without the constant. Acceptable for this spec; leave the
follow-up on the books.

### 3. Corpus is all-`jq_equivalent` (observation)

Every case in `cases.json` is `jq_equivalent: true`, so the `filter(|c|
c.jq_equivalent)` in ac-06 is currently a no-op. That's fine — the field is the
seam that lets future non-equivalent cases (e.g. reproducibility-only fixtures)
live in the same corpus without breaking parity. No action.

## Why APPROVE not REQUEST_CHANGES

The gate (ac-02, true passthrough) holds cleanly: the program is never routed
through an AST round-trip that reformats it before the reviewer sees it — it is
handed to the loader as raw `code` and echoed verbatim, with a byte-equality
test guarding it. The two parity claims are correctly *split* and each
independently tested, and the reference-parity test genuinely runs here rather
than silently skipping. Finding 1 is a wording/scope mismatch on a non-gate AC,
not a correctness or legibility failure, and the output contract it guards
(valid NDJSON) is met. Build is warning-free; the survey path is untouched.

Record findings 1 and 2 against card 0003 as follow-ups; ship this spec.
